WARNUserService.getAllUsers — неправильная проверка
findAll() возвращает массив, который никогда не будет null — проверка бессмысленная. При пустой БД вернётся [], а не null.
if (!users) {  ← никогда не будет true
  throw new NotFoundException('...');
}


WARNOrderController — несоответствие интерфейса IAuthenticatedUser
В OrderController объявлен interface IAuthenticatedUser { userId: string }, но CurrentUser декоратор возвращает AccessTokenPayload с полем sub, а не userId. Код не сломается, но типы неправильные.
interface IAuthenticatedUser {
  userId: string;  ← но JWT payload содержит 'sub', не 'userId'!
}


WARNAllExceptionsFilter — потеря деталей validation errors
При ошибке валидации (400) class-validator возвращает массив сообщений. Фильтр приводит exceptionResponse.message к одной строке — пользователь видит только первую ошибку.
message = (exceptionResponse as { message: string }).message
// Если message — массив ['email must be...', 'password too short'] → только первое!

WARNCI — postgres:15 в CI vs postgres:16 в docker-compose
В ci.yml используется postgres:15-alpine, а в docker-compose.yml — postgres:16. Разные версии PostgreSQL могут давать разное поведение.

WARNE2E — login возвращает 201, а не 200
В auth.e2e-spec.ts тест ожидает 201 для POST /auth/login. Семантически правильно — 200 OK (пользователь не создаётся, только аутентифицируется). Нужно добавить @HttpCode(200) на login endpoint.
